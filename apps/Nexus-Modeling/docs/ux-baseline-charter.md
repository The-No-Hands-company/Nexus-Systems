# UX Baseline Charter

This document defines the Month 1 baseline UX constraints for Nexus Modeling.

## Core Principle

Nexus Modeling must be simple by default and powerful by intent.

## First-Launch Baseline

The default product experience must:

1. Avoid presenting a wall of advanced tools on first launch.
2. Provide a guided introduction path.
3. Start from a lean creation surface centered on starter primitives and core actions.

## Starter Tool Policy

The starter surface should prioritize:

1. 6 to 10 foundational primitives.
2. Core modeling actions such as inset, extrude, move, rotate, and scale.
3. Contextual visibility of tools based on selection and current mode.

## Expert Expansion Policy

Advanced capability must remain available through:

1. Searchable settings/preferences.
2. Profile-based workspace customization.
3. User-defined quick access surfaces and advanced panel exposure.
4. Domain-specific advanced tools that are opt-in rather than always-on.

## Non-Negotiable UX Rules

1. New features must not regress onboarding clarity.
2. Advanced tools should surface by context, profile, or explicit user action.
3. The default workspace must be understandable to a first-time user.
4. Expert customization must not require source modification.

## Acceptance Checks

Month 1 UX baseline is satisfied only when these checks are true:

1. A documented first-launch flow exists.
2. Starter primitives and core actions are clearly identified as the default entry set.
3. Advanced tool surfacing is documented as profile/customization-driven.
4. Roadmap and product docs reference progressive disclosure as a mandatory design rule.
